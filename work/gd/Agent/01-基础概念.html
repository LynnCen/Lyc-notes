一、什么是 AI Agent？ ​

1.1 传统软件 vs AI Agent ​

让我们从最基本的问题开始：AI Agent 和传统软件有什么本质区别？

传统软件（规则驱动）：

python
def process_request(user_input):
    if user_input == "天气":
        return get_weather()
    elif user_input == "时间":
        return get_time()
    else:
        return "不理解"

AI Agent（智能驱动）：

python
def ai_agent(user_input):
    # 1. LLM 理解用户意图
    intent = llm.understand(user_input)
    
    # 2. 规划需要调用的工具
    tools = llm.plan_tools(intent)
    
    # 3. 执行工具调用
    results = execute_tools(tools)
    
    # 4. LLM 整合结果并回复
    return llm.generate_response(intent, results)

2.2 统一消息格式的必要性 ​

💡 核心洞察（来自实践）
"愈发觉得前期不应该把 LLM 的消息格式当作业务格式来使用。面向业务领域的消息格式，势必要考虑状态和交互的问题，否则无法通过一个源来回溯所有的状态。"

为什么需要统一？

  1. 多模型支持
    我们需要支持多种 LLM（GPT、Claude、Doubao、Qwen），每个模型的消息格式都不同

  2. 业务扩展
    需要记录业务信息：

    • 用户选择的技能 ID
    • 稿豆扣除信息
    • 本地消息 ID（前端关联）
    • 附件信息

  3. 状态管理
    需要支持：

    • 中断（稿豆不足、Token 过期）
    • 恢复（用户充值后继续）
    • 停止（用户主动停止）

  4. 可追溯性
    需要完整记录：

    • 每条消息的创建时间
    • 消息之间的关联关系
    • 工具调用的完整链路

三、消息格式深入原理 ​

3.1 消息的本质 ​

从计算机科学的角度看，消息就是状态机的状态转移：

State_0 (初始状态)
   ↓ [user message]
State_1 (等待 LLM 响应)
   ↓ [assistant message]
State_2 (LLM 回复完成)
   ↓ [function call]
State_3 (等待工具执行)
   ↓ [function response]
State_4 (工具执行完成)
   ↓ [assistant message]
State_5 (最终回复)

4.3 状态码设计原则 ​

为什么状态码是负数？

正数：业务正常流程

  -1002: 稿豆不足
  -1003: 调用超限
  -1004: Token 过期
  -1005: 心跳
  -1006: 内容风险
  -1007: 安全预警

-2xxx：前端生成的状态消息
  -2001: 恢复执行

  3. 防止冲突：后端和前端不会有相同的状态码

六、会话 (Thread) 的本质 ​

6.1 Thread 是什么？ ​

Thread（线程/会话）是一个独立的对话上下文。

类比：

特点：

  • 每个 Thread 有唯一 ID
  • Thread 内的消息按时间顺序排列
  • Thread 之间相互独立

七、核心设计原则 ​

7.1 不可变性 (Immutability) ​

原则：消息一旦创建，不可修改。

为什么？

  1. 审计追踪
    可以追溯每条消息的完整历史

  2. 并发安全
    多个请求同时读取消息时不会冲突

  3. 状态回溯
    可以回到任意历史状态

实现：

python
# ✅ 正确：创建新消息
def add_message(thread_id, content):
    message = create_message(thread_id, content)
    save_message(message)
    return message

# ❌ 错误：修改已有消息

    message = get_message(message_id)
    message.content = new_content  # ❌ 不允许
    save_message(message)

7.2 单一数据源 (Single Source of Truth) ​

原则：所有状态都通过消息记录。

为什么？

  1. 完整性
    通过消息序列可以完全恢复状态

  2. 一致性
    不会出现状态不一致的情况

  3. 可追溯
    所有变化都有记录

示例：

python
# ✅ 正确：通过消息记录状态
def get_conversation_state(thread_id):
    messages = get_messages(thread_id)
    state = {
        "llm_calls": count_llm_calls(messages),
        "tool_calls": count_tool_calls(messages),
        "is_interrupted": has_status_message(messages, -1002),
    }
    return state

# ❌ 错误：单独维护状态
conversation_states = {
    "thread_123": {"llm_calls": 5, "tool_calls": 2}
}

7.3 幂等性 (Idempotency) ​

原则：相同的输入多次执行产生相同的结果。

为什么？

  1. 重试安全
    网络失败时可以安全重试

  2. 分布式友好
    多个节点可以处理相同请求

实现：

python
# 使用 message_id 保证幂等
def create_message(message_id, content):
    # 检查是否已存在
    if exists(message_id):
        return get_message(message_id)
    
    # 不存在则创建
    message = Message(id=message_id, content=content)
    save_message(message)
    return message

八、总结 ​

8.1 核心概念回顾 ​

  1. AI Agent = LLM + 工具调用 + 状态管理 + 任务编排

  2. 消息是状态机的状态转移

  3. 统一消息格式的必要性：

    • 支持多模型
    • 业务扩展
    • 状态管理
    • 可追溯性

  4. 5 种消息角色：

    • system：系统指令
    • user：用户输入
    • assistant：AI 回复
    • function：工具结果
    • status：状态消息

  5. 工具调用的本质：让 LLM 输出结构化指令

  6. Thread 是独立的对话上下文

8.2 设计原则 ​

  • ✅ 不可变性：消息不可修改
  • ✅ 单一数据源：状态通过消息记录
  • ✅ 幂等性：可以安全重试
