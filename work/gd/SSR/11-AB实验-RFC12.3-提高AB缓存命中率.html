RFC 12.3: 提高 AB 缓存命中率 ​

文档来源
Confluence: https://doc.huanleguang.com/pages/viewpage.action?pageId=438705960

文档概述 ​

本 RFC 专注于解决 AB 实验导致的缓存碎片化问题，通过优化缓存策略显著提升登录态下的页面缓存命中率。

核心目标：

  • 📈 提升登录态缓存命中率：从 10-15% → 40-50%
  • 🎯 解决缓存碎片化问题
  • ⚡ 改善页面性能指标（TTFB、LCP）
  • 🔧 区分公共实验和私有实验

相关文档：

  • 前置：RFC 12.1: AB 实验与缓存策略改进
  • 前置：RFC 12.2: AB 实验数据获取优化

一、背景与问题 ​

1.1 性能下降现象 ​

数据观察：从 2025 年第 5 周开始，首页和模板页等 SSR 页面的性能持续下降

性能指标趋势：

性能指标第3周第4周第5周第6周第7周第8周第9周
首页 TTFB611 ms798 ms1070.5 ms1027 ms1104.8 ms1203.6 ms1201.6 ms
首页 LCP1556.5 ms1695.5 ms2145.3 ms1956.3 ms2147.5 ms2239.9 ms2244 ms

趋势分析：

TTFB: 611ms → 1201ms (⬆️ 96%)
LCP:  1556ms → 2244ms (⬆️ 44%)

影响：

  • 🔴 TTFB 增加近 100%
  • 🔴 LCP 从优秀（< 1.5s）降到不及格（> 2s）
  • 🔴 用户体验明显下降

1.2 问题分析 ​

分析步骤 ​

步骤 1：关联分析

观察：性能下降 和 缓存命中率下降 趋势一致

结论：怀疑缓存命中率影响了性能

实验：第 11 周，关闭首页的 AB 缓存 key
   ↓
观察：
   - 缓存命中率上升 ✅
   - TTFB 和 LCP 下降 ✅
   ↓
结论：AB 缓存 key 确实影响性能

步骤 3：详细分析

维度缓存命中率结论
未登录态~60%🟢 可接受
已登录态~10-15%🔴 严重问题
整体~35%🟡 需改善

关键发现：关闭 AB 缓存 key 后，登录态的缓存命中率提升明显

根本原因 ​

未登录态：

缓存 Key = hash(URL + ab-test-id)

特点：
- 用户之间可以共享缓存
- 缓存命中率相对较高（~60%）
- AB 调整会影响，但影响面广，新缓存快速积累

缓存 Key = hash(URL + userId + orgId + ab-test-id)

问题：
1. userId 和 orgId 导致每个用户独立缓存
2. ab-test-id 进一步分裂缓存
3. AB 调整后，所有用户的缓存都失效
4. 用户数多，缓存碎片化严重

结果：
- 缓存命中率极低（~10-15%）

二、解决方案探索 ​

2.1 方案一：登录态下移除 ab-test-id ​

核心思路 ​

理论依据：

userId + orgId 已经标识了用户的唯一性
   ↓
同一用户在同一时刻只会命中一种 AB 组
   ↓
理论上不需要 ab-test-id 区分缓存

function getCacheKey(url, userId, orgId, abTestId, isLoggedIn) {
  if (isLoggedIn) {
    // 登录态：不使用 ab-test-id
    return hash(url + userId + orgId);
  } else {
    // 未登录态：使用 ab-test-id
    return hash(url + abTestId);
  }
}

方案流程设计 ​

区分登录态：

javascript
// 生成 ab-test-id
function generateABTestId(abData, isLoggedIn) {
  if (isLoggedIn) {
    // 登录态：只使用公共实验
    const commonExperiments = filterCommonExperiments(abData);
    return hash(JSON.stringify(commonExperiments));
  } else {
    // 未登录态：使用所有实验
    return hash(JSON.stringify(abData));
  }
}

// 过滤公共实验
function filterCommonExperiments(abData) {
  const commonExperiments = {};
  
  for (const [key, value] of Object.entries(abData)) {
    // 检查实验的 tag
    if (value.tags && value.tags.includes('common')) {
      commonExperiments[key] = value;
    }
  }
  
  return commonExperiments;
}

Redis 存储策略：

javascript
// 登录态存储
async function storeABDataForLoggedIn(abData, userId) {
  // ab-test-common-id: 只包含公共实验的 hash
  const commonExperiments = filterCommonExperiments(abData);
  const abTestCommonId = hash(JSON.stringify(commonExperiments));
  
  // Redis Key 包含 userId，避免不同用户冲突
  const redisKey = `ab:${userId}:${abTestCommonId}`;
  
  // Value 仍然是完整的 AB 数据
  await redis.setex(redisKey, 1800, JSON.stringify(abData));
  
  return abTestCommonId;
}

CSR 数据使用 ​

javascript
// 客户端 AB SDK
class ABClient {
  constructor() {
    this.init();
  }
  
  init() {
    // 优先使用 SSR 注入的数据
    if (window.GD_AB_DATA) {
      this.abData = window.GD_AB_DATA;
      this.syncToLocalStorage();
      return;
    }
    
    // 检查 localStorage 缓存
    const cachedData = this.getFromLocalStorage();
    if (cachedData && this.isValid(cachedData)) {
      this.abData = cachedData.data;
      return;
    }
    
    // 降级：重新获取
    this.fetchABData();
  }
  
  syncToLocalStorage() {
    const abTestId = this.getCookie('x-ab-test-id');
    localStorage.setItem('ab-experiments', JSON.stringify({
      id: abTestId,
      data: this.abData,
      timestamp: Date.now()
    }));
  }
  
  isValid(cachedData) {
    // 检查 ab-test-id 是否一致
    const currentId = this.getCookie('x-ab-test-id');
    return cachedData.id === currentId;
  }
  
  // 埋点上报使用完整数据
  trackEvent(eventName, extra = {}) {
    track(eventName, {
      ...extra,
      ab_experiments: this.abData // 完整的 AB 数据
    });
  }
  
  // 接口请求携带 AB 标识
  async apiRequest(url, options = {}) {
    return fetch(url, {
      ...options,
      headers: {
        ...options.headers,
        'X-AB-Experiments': JSON.stringify(this.abData)
      }
    });
  }
}

  const commonExperiments = {};

5.2 降级方案 ​

javascript
// 配置开关
const CONFIG = {
  useCommonExperimentsOnly: true,  // 主开关
  fallbackToAllExperiments: true   // 降级开关
};

// Worker 层降级
async function generateABTestId(abData, isLoggedIn) {
  if (!CONFIG.useCommonExperimentsOnly) {
    // 降级：使用所有实验（旧逻辑）
    return hash(JSON.stringify(abData));
  }
  
  try {
    if (isLoggedIn) {
      const commonExperiments = filterCommonExperiments(abData);
      
      if (Object.keys(commonExperiments).length === 0) {
        // 没有公共实验，降级
        console.warn('No common experiments found, fallback');
        if (CONFIG.fallbackToAllExperiments) {
          return hash(JSON.stringify(abData));
        }
      }
      
      return hash(JSON.stringify(commonExperiments));
    } else {
      return hash(JSON.stringify(abData));
    }
  } catch (error) {
    console.error('Generate AB test ID failed, fallback', error);
    return hash(JSON.stringify(abData));
  }
}

六、总结 ​

6.1 核心价值 ​

问题解决：

  1. ✅ 缓存碎片化

    • 从 2^n 降到 2^m (m << n)
    • 登录态缓存命中率提升 3-4 倍

  2. ✅ 性能改善

    • TTFB 降低 40%+
    • LCP 降低 30%+
    • 回到优秀水平（< 1.5s）

  3. ✅ 成本节省

    • SSR 渲染次数减少 20%
    • 服务器资源节省 30%

  4. ✅ 体验提升

    • 页面加载更快
    • 公共组件保持一致

6.2 关键要点 ​

技术要点：

  1. ✅ 区分公共实验和私有实验

    • 公共实验：SSR 渲染的共享组件
    • 私有实验：页面特定或 CSR 组件

  2. ✅ 登录态和未登录态分别处理

    • 未登录：所有实验参与（用户可共享缓存）
    • 已登录：仅公共实验参与（减少碎片化）

  3. ✅ 完整数据存储

    • Redis 存储完整 AB 数据
    • CSR 可获取所有实验信息
    • 埋点和接口正常工作

  4. ✅ 用户隔离

    • Redis Key 包含 userId
    • 避免不同用户数据冲突

6.3 最佳实践建议 ​

实验设计：

  1. ⭐ 控制公共实验数量

    • 建议：≤ 3 个同时进行
    • 避免过度碎片化

  2. ⭐ 优先使用私有实验

    • 页面特定功能用私有实验
    • 不影响缓存

  3. ⭐ 实验生命周期管理

    • 及时下线完成的实验
    • 避免长期占用

监控和运维：

  1. ✅ 实时监控缓存命中率
  2. ✅ 定期审查公共实验列表
  3. ✅ 评估实验对缓存的影响
