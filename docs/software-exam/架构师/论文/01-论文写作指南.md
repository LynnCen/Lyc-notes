# 系统架构设计师论文写作指南

## 一、考试要求

### 1.1 论文考核目的

1. **检查实践经验**：完全不具备实践经验的人很难胜任系统架构设计师的工作
2. **检查分析与解决问题能力**：能把握系统关键因素，发现和分析问题，根据实际情况提出架构设计方案
3. **检查表达能力**：文档编写能力，抓住要点、重点突出、用词准确

### 1.2 字数要求

| 部分 | 字数要求 | 建议字数 |
|------|---------|---------|
| 摘要 | 300～400字 | 350字左右 |
| 正文 | 2000～3000字 | 2500字左右 |

**注意事项**：
- 摘要少于120字，论文将"给予不及格"
- 没有写摘要或摘要过于简略将扣5～10分
- 正文超过3000字答题纸写不完

### 1.3 答题纸格式

- 第1页：摘要16行（400字）+ 正文12行（300字）
- 第2～4页：每页36行（900字）
- 每12行有字数提示（300、600、900）

---

## 二、摘要写作

### 2.1 摘要模板

**模板一**：
```
本文讨论……系统项目的……(论文主题)。该系统……(项目背景、简单功能介绍)。
在本文中首先讨论了……(技术、方法、工具、措施、手段)，最后……(不足之处/如何改进、特色之处、发展趋势)。
在本项目的开发过程中，我担任了……(作者的工作角色)。
```

**模板二**：
```
根据……需求(项目背景)，我所在的……组织了……项目的开发。该项目……(项目背景、简单功能介绍)。
在该项目中，我担任了……(作者的工作角色)。我通过采取……(技术、方法、工具、措施、手段)，
使该项目圆满完成，得到了用户们的一致好评。但现在看来，……(不足之处/如何改进、特色之处、发展趋势)。
```

**模板三**：
```
……年……月，我参加了……项目的开发，担任……(作者的工作角色)。该项目……(项目背景、简单功能介绍)。
本文结合作者的实践，以……项目为例，讨论……(论文主题)，包括……(技术、方法、工具、措施、手段)。
```

### 2.2 摘要要点

- 概括反映正文全貌，引人入胜
- 不要只谈大道理，要有实质性内容
- 包含项目背景、目标、技术手段、角色、成效
- 一般不要在摘要中"戴帽子"（字数不够时可加50字左右）

## 三、正文写作技巧

### 3.1 以自我为中心

- 体现实际经验，不要罗列课本内容
- 条理性说明实际经验
- 写明项目开发体制和规模
- 明确"我"的工作任务和所起的作用
- 以"我"在项目中的贡献为重点

### 3.2 站在架构师角度

- 从全局角度把握架构设计的优缺点
- 考虑设计方法、过程、接口设计问题
- 不要专注于具体实现细节

### 3.3 忠实于论点

- 正确理解题意，仔细阅读论文试题要求
- 根据正确的题意提取论点加以阐述
- 就试题的问题进行展开，不要节外生枝

### 3.4 条理清晰，开门见山

- 列出提纲（打算谈几个方面、怎么做的、收效如何）
- 不要试图覆盖论文题目的全部内涵
- 重点放在汇报自己在项目中所做的与论题相关的工作

### 3.5 标新立异，要有主见

- 写一些个人的见识和体会
- 不要拘泥于教科书或常规思维

### 3.6 图文并茂

- 紧要地方画草图表示，能收到奇效
- 图形不要画得太"草"，也不宜过大
- 线条、箭头等保持整洁

### 3.7 首尾一致

- 开头与结尾互相呼应
- 言词的意思忌途中变卦
- 检查错字、漏字

## 四、常见问题

| 问题 | 说明 |
|------|------|
| 走题 | 不认真阅读试题的3个问题，按照三段论的方式写 |
| 字数不够 | 摘要<300字或正文<2000字 |
| 字数偏多 | 正文>3000字，答题纸写不完 |
| 摘要归纳欠妥 | 摘要是正文的压缩，不是帽子性语句 |
| 深度不够 | 措施太多但都没有深入，选2-3个深入展开 |
| 泛泛而谈 | 罗列书刊杂志知识点，可信性不强 |
| 口语化太重 | 使用书面语言，用"我们"代替一些"我" |
| 文字表达能力差 | 平时多读文章，多写文档 |
| 缺乏主题项目 | 必须说明某年某月参加的某个具体项目 |
| 项目年代久远 | 主题项目应该是近3年内完成的 |
| 结构死板 | 不要全文充满数字条目 |
| 段落太长 | 每个自然段最好不超过8行 |

## 五、论文三段式结构

### 第一部分：项目背景与职责（约20%，300-400字）

**包含内容**：
- 项目规模：业务规模、技术栈、团队规模
- 项目背景：为什么要采用该技术/架构
- 职责范围：你在项目中的角色（架构设计、核心模块开发等）
- 关键业务需求与技术挑战

### 第二部分：技术原理与方法（约35%，600-800字）

**包含内容**：
- 核心概念详解（官方定义+自己理解）
- 技术方案的关键要素
- 为什么选择这种方案（对比分析）
- 工作流程与机制说明
- 必要的图表（架构图、流程图）

### 第三部分：实施与效果（约45%，1200-1500字）

**包含内容**：
- 具体实施过程（各阶段做了什么）
- 遇到的问题（3-4个代表性问题）
- 解决方案（技术手段、实施方法）
- 效果验证（定量指标、定性收获）
- 经验教训与启示

---

## 六、时间分配建议

| 阶段 | 时间 | 内容 |
|------|------|------|
| 第一部分 | 15分钟 | 项目背景，300-400字 |
| 第二部分 | 30分钟 | 理论阐述，600-800字 |
| 第三部分 | 60分钟 | 实践论述，1200-1500字 |
| 检查修改 | 15分钟 | 检查错字、逻辑 |
| **总计** | **2小时** | **全文2200-2500字** |

---

## 七、答题自查清单

- [ ] 是否能用一张图讲清架构与链路？
- [ ] 是否有2个可量化的效果数字（前后对比）？
- [ ] 是否准备1个"权衡取舍"的关键决策案例？
- [ ] 是否准备"降级/熔断/回滚/演练"的闭环证据？
- [ ] 是否准备"数据质量/一致性/安全合规"的证明材料？
- [ ] 摘要是否涵盖正文的全部要点？
- [ ] 正文是否回答了题目的三个问题？
